# 提示词③：代码审计与重构

**用途**：随时调用。AI 初版跑通不代表能用，用它做审计重构。

## 核心审计目标

必须系统性地排查以下 AI 编程中极易出现的典型缺陷，并严格按照指定的优化方向执行重构。

---

## 1. 结构与模块化坍塌

**风险点（AI 常犯）**：巨石代码与超长文件。将所有逻辑堆砌在单个文件中（如超过 400-500 行的 Vue 组件或庞大的 Python 服务类）。

**重构方向**：
- 前端：将复杂大组件拆分为单一职责的子组件，视图层与逻辑层彻底分离，复杂状态管理抽离至 composables 或 stores
- 后端：严格遵循 Controller (Router) → Service (业务逻辑) → Repository (数据操作) 的分层架构

## 2. 零硬编码与幻觉隔离

**风险点（AI 常犯）**：将正则、状态字典、业务阈值直接写死在逻辑流中；生造依赖或重复造轮子。

**重构方向**：
- 所有业务规则、正则、状态值剥离至全局配置表、数据库或环境变量
- 修改前全局检索项目上下文，强制复用现有的基础架构代码

## 3. 逻辑短路与伪容错

**风险点（AI 常犯）**：滥用裸 try-catch，遇到未预见的数据结构时用空值敷衍兜底，导致真实底层错误被吞噬。

**重构方向**：
- 移除所有静默失败逻辑
- 建立显式、精确的错误冒泡与捕获机制
- 强制在全链路流转节点使用强校验（Pydantic v2 / Zod）

## 4. 数据层级与状态失控

**风险点（AI 常犯）**：在处理多级嵌套数据时，为图简便直接丢弃层级关系或父级标识。

**重构方向**：
- 强制保留并校验所有数据层级标识，执行合并而非覆盖
- 严格遵循 Vue 3 响应式最佳实践，正确使用 toRefs、computed

## 5. 性能陷阱与并发灾难

**风险点（AI 常犯）**：在异步框架的协程中运行重型 CPU 计算或同步 I/O；在循环内部执行数据库查询或 API 请求。

**重构方向**：
- 核心事件循环必须保持非阻塞（CPU 密集型任务交由后台 Worker 处理）
- 强制将循环内的单次查询重构为批量查询
